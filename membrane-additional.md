# Membrane Engine — Additional Context
> 补充文档，记录设计者的核心想法与产品方向，供 Claude AI 审阅参考

---

## 项目背景与命名

引擎代号 **Membrane**，取"薄膜"之意——薄、透明，是边界也是连接。玩家看到的只是膜的一面，膜的另一面是 AI 可操作的结构化数据。

Tagline：**The Runtime Layer for AI-Native Games**

---

## 核心设计哲学

### 引擎即协议层，而非工具链

传统引擎（Unity、虚幻）的价值在于给人用的工具链——编辑器、材质系统、蓝图。Membrane 的判断是：AI 时代这一层将被 AI 工具替代，引擎本身只需要是一个**薄的运行时 + 连接器协议**。

类比：**Notion 之于文档**。Notion 的核心不是富文本编辑器，而是结构化内容 + 连接器。对人呈现漂亮的格式，对 AI 暴露可操作的数据。Markdown 是人机共读的最优格式——信息密度高、歧义低、AI 理解力强。

### 三层分离

```
内容管理层（Notion-like 网页工具）
    ├── AI 图像工具（生成 2D 素材/UI）
    ├── AI 代码工具（生成玩法逻辑）
    ├── 其他 AI 连接器（MCP 协议接入）
    └── 输出：结构化场景数据（AI 友好格式）
           ↓
运行时层（Membrane Runtime）
    ├── 读取结构化数据并执行
    ├── ECS + WebGL + WASM
    └── 对玩家呈现游戏界面
           ↓
发布层
    └── 微信小游戏打包发布
```

**运行时本身不需要知道内容是人写的还是 AI 生成的，只认数据格式。**

## 数据格式观点

2D 小游戏的本质极其简单：

```
场景 = 图层列表
  每个图层 = 一堆精灵
    每个精灵 = 图片引用 + 坐标/旋转/缩放 + 标签

逻辑 = 事件表
  on(触发条件) → do(动作列表)
```

内部格式示例：
```json
{
  "scene": "level_01",
  "layers": [
    {
      "name": "entities",
      "sprites": [
        { "img": "player.png", "x": 100, "y": 200, "tags": ["player", "physics"] },
        { "img": "coin.png",   "x": 300, "y": 200, "tags": ["collectible"] }
      ]
    }
  ],
  "events": [
    { "on": "tag:player touch tag:collectible", "do": "destroy:target, score:+1" }
  ]
}
```

具体格式是实现细节，随时可改，不是瓶颈。**格式的稳定性是核心资产，上下两层能否解耦取决于此。**

## 开发路径规划

### 阶段一：先做运行时（当前重点）

- 目标平台：微信小游戏（平台约束强迫做薄工具链，是天然的验证场）
- 先借助已有小游戏（Cocos Creator / Unity 生产的）
- 导出为 Membrane 内部格式，跑通运行时
- 导入粒度：以场景级为主（A 方案），因为早期最现实

Cocos `.scene` 文件本身是 JSON，解析器工作量低。难点在于脚本组件翻译为事件表。

### 阶段二：边做工具

- 工具不集成为 Cocos Creator / Unity 那样的重 IDE
- 而是一个 **Notion-like 的内容管理网页**
- 对人：提供漂亮可读的内容组织界面
- 对 AI：暴露 MCP 接口，AI 可直接读写游戏数据

### 阶段三：对话即创作

- 工具暴露 MCP 接口后，对话就能驱动内容生产
- "做一个跑酷游戏，障碍物越来越快" → AI 生成场景数据 → 运行时直接执行
- AI 介入时机：**内容生产端**（编辑期），运行时保持确定性

---

## 为什么先从微信小游戏开始

- 帧率压力相对 PC / 原生移动端低，容错空间更大
- 平台约束（主包 ≤ 4MB，禁多线程，WXWebAssembly）天然逼出薄运行时架构
- 市场验证快，发布门槛低
- 设计者当前经验积累更适合此平台

原生移动端 / PC 的帧率维护更复杂，留到后续阶段。

---

## AI 在引擎中的定位

Membrane 中 AI 的角色是**内容生产者**，不是运行时决策者：

| 层级 | AI 角色 | 说明 |
|------|---------|------|
| 内容生产端 | 主力 | 生成素材、场景、逻辑描述 |
| 运行时逻辑层 | 可选 | NPC 行为等，后期探索 |
| 运行时调度层 | 不介入 | 保持确定性和性能可预测 |

---

## 一句话总结设计者的核心直觉

> 游戏引擎应该像 Notion 一样：对人是漂亮的界面，对 AI 是一堆可操作数据。工具链交给 AI，引擎只做最薄的那一层。
